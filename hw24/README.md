# Принципи рефакторингу, виявлення та усунення code smells
# Контрольні питання
## 1. Що таке рефакторинг і навіщо він потрібен?
Рефакторинг — це процес покращення внутрішньої структури програмного коду без зміни його зовнішньої поведінки. Основна мета рефакторингу полягає у підвищенні читабельності коду, спрощенні його підтримки та підготовці системи до подальшого розширення.
Рефакторинг потрібен для:
- зменшення складності коду;
- підвищення якості архітектури;
- полегшення тестування;
- спрощення внесення нових змін;
- зменшення ризику виникнення помилок у майбутньому.
## 2. Назвіть 5 найпоширеніших code smells та способи їх усунення
### Long Method  
Метод містить занадто багато логіки.
**Спосіб усунення:** Extract Method
### Large Class  
Клас має занадто багато відповідальностей.
**Спосіб усунення:** Extract Class
### Duplicate Code  
Однакова логіка повторюється в кількох місцях.
**Спосіб усунення:** винесення спільного коду в окремий метод
### Long Parameter List  
Метод має занадто багато параметрів.
**Спосіб усунення:** Introduce Parameter Object
### Magic Numbers  
У коді використовуються числа без пояснення їхнього значення.
**Спосіб усунення:** Extract Constant
## 3. Що таке God Object і як його виправити?
God Object — це клас, який містить надто багато методів, полів та відповідальностей. Такий клас стає центральною точкою системи та порушує принцип єдиної відповідальності.
Виправлення полягає у розділенні великого класу на декілька менших класів, кожен із яких виконує окрему функцію.
## 4. Чому важливо мати тести перед рефакторингом?
Тести дозволяють перевірити, що після змін поведінка програми не змінилася.
Без тестів існує ризик:
- випадково зламати логіку;
- не помітити нову помилку;
- отримати нестабільний код.
## 5. Що таке цикломатична складність і як вона впливає на якість коду?
Цикломатична складність — це метрика, яка показує кількість незалежних шляхів виконання програми.
Чим більше умовних операторів і циклів у коді, тим вища складність.
Висока складність означає:
- важче тестувати код;
- складніше підтримувати;
- більший ризик помилок.

# Аналіз лабораторної роботи
У лабораторній роботі реалізовано систему Smart Home з двома підходами:
- початкова реалізація (Bad)
- покращена реалізація після рефакторингу
# 1. Code Smell: Tight Coupling
## Проблема
У класі SmartHomeCentralBad створення об'єктів відбувається всередині класу, що створює жорстку залежність між компонентами.
## Код до рефакторингу
```csharp
class SmartHomeCentralBad
{
    private LightControllerBad _light;
    private MusicPlayerBad _music;
    private SecurityAlarmBad _alarm;

    public SmartHomeCentralBad()
    {
        _light = new LightControllerBad();
        _music = new MusicPlayerBad();
        _alarm = new SecurityAlarmBad();
    }
}
```
## Код після рефакторингу

```csharp
class SmartHomeCentral
{
    private readonly ILight _light;
    private readonly IMusicPlayer _music;
    private readonly ISecurityAlarm _alarm;

    public SmartHomeCentral(
        ILight light,
        IMusicPlayer music,
        ISecurityAlarm alarm)
    {
        _light = light;
        _music = music;
        _alarm = alarm;
    }
}
```
## Застосована техніка рефакторингу
Dependency Injection
# 2. Code Smell: Відсутність абстракції
## Проблема
У початковій версії кожен клас працює окремо без єдиного контракту.
## Код до рефакторингу
```csharp
class LightControllerBad
{
    public void TurnOnLight()
    {
        Console.WriteLine("Світло увімкнено (Bad)");
    }
}
```
## Застосована техніка рефакторингу
Extract Interface
## Код після рефакторингу
```csharp
interface ILight
{
    void TurnOn();
}

class LightController : ILight
{
    public void TurnOn()
    {
        Console.WriteLine("Світло увімкнено");
    }
}
```
# 3. Code Smell: Poor Scalability
## Проблема
При додаванні нового пристрою потрібно змінювати центральний клас.
## Код до рефакторингу
```csharp
public void StartEveningMode()
{
    _light.TurnOnLight();
    _music.PlayMusic();
    _alarm.ActivateAlarm();
}
```
## Код після рефакторингу
```csharp
public void StartEveningMode()
{
    _light.TurnOn();
    _music.Play();
    _alarm.Activate();
}
```
## Застосована техніка рефакторингу
Programming to Interface
### Чому рефакторинг без тестів є ризикованим
Якщо випадково прибрати один із викликів:
```csharp
public void StartEveningMode()
{
    _light.TurnOn();
    _music.Play();
}
```
Сигналізація більше не активується.
Тести дозволяють швидко виявити таку помилку.
Висновок

У лабораторній роботі виявлено три типові проблеми: жорсткі залежності, відсутність абстракцій,
слабка масштабованість.
Після рефакторингу код став більш гнучким, читабельним і готовим до подальшого розвитку.
