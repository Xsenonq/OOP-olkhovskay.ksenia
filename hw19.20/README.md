#  Lab 19 - Composite & Decorator
## Composite і Decorator: концепції та компроміси.
**Composite** - структурний патерн проєктування, що дозволяє будувати деревоподібні структури типу частина-ціле та працювати з ними як з єдиною колекцією.
Основна ідея - **уніфікований інтерфейс**.  
Клієнтський код не розрізняє окремі об’єкти (Leaf) і групи об’єктів (Composite).
###  Як це працює
- Усі елементи реалізують один інтерфейс (Component)
- Composite може містити Leaf і інші Composite
- Операції виконуються рекурсивно
- Клієнт працює з усією структурою однаково
Дерево поводиться як одна колекція.
###  Реальні сценарії
1. Файлова система (файли / папки)  
2. UI компоненти (контейнери / елементи)  
3. Організаційні структури  
4. Scene graph у game development
## Переваги та недоліки
### Переваги
- Уніфікація роботи з ієрархіями
- Спрощення клієнтського коду
- Розширюваність
- Природна рекурсивність
###  Недоліки
- Складніше обмежувати типи дочірніх елементів
- Інтерфейс може бути занадто загальним
- Порожні методи у Leaf
- Складніший дебаг
## Decorator (коротко)
**Decorator** - патерн, що дозволяє динамічно додавати функціональність об’єкту без зміни його класу.
###  Коли застосовують
- Додавання поведінки під час виконання
- Комбінування функціональності
- Альтернатива великій кількості підкласів
###  Переваги
- Гнучкість
- Відповідність OCP
- Уникнення “class explosion”
###  Недоліки
- Багато дрібних об’єктів
- Порядок декораторів важливий
## Висновок  19
Composite дозволяє працювати з ієрархічними структурами як з єдиною колекцією, що значно спрощує архітектуру системи.
Decorator доповнює його, надаючи механізм гнучкого розширення поведінки без зміни існуючого коду.
Разом ці патерни забезпечують високу гнучкість та масштабованість.
#  Lab 20 - Adapter, Facade, Proxy: інтеграція й спрощення.
Ці патерни вирішують різні архітектурні задачі інтеграції та спрощення.
- **Adapter** - інтеграція несумісних інтерфейсів  
- **Facade** - спрощення складних систем  
- **Proxy** - контроль доступу та оптимізація  
## Adapter
**Adapter** перетворює інтерфейс одного класу на інший, який очікує клієнт.
 Патерн-перекладач.
###  Реальні сценарії
1. Інтеграція сторонніх API  
2. Робота з legacy кодом  
3. Конвертація форматів (XML -- JSON)  
4. Підключення SDK
###  Що вирішує
- Несумісні інтерфейси  
- Повторне використання коду  
## Facade
**Facade** надає спрощений інтерфейс до складної підсистеми.
 Один виклик - багато внутрішніх операцій.
###  Реальні сценарії
1. Startup системи / engine  
2. Home theater  
3. Backend orchestration  
4. Сервіс “оформити замовлення”
###  Що вирішує
- Зменшення складності
- Інкапсуляцію підсистеми
- Слабке зв’язування
## Proxy
**Proxy** - заступник, який контролює доступ до реального об’єкта.
 Додає логіку без зміни RealSubject.
###  Реальні сценарії
1. Lazy loading  
2. Кешування  
3. Контроль доступу  
4. Remote proxy  
5. ORM lazy loading
###  Що вирішує
- Контроль доступу
- Оптимізацію
- Логування
- Безпеку
## Ключова відмінність
 Adapter - Змінює інтерфейс 
 Facade - Спрощує використання 
 Proxy - Контролює доступ 
 Просте правило:
- Adapter = перекладач  
- Facade = спрощувач  
- Proxy = посередник  
## Висновок 20
Adapter, Facade і Proxy - базові патерни інтеграції систем.
- Adapter дозволяє поєднувати несумісні компоненти  
- Facade приховує складність підсистем  
- Proxy додає контроль, оптимізацію та безпеку  
Використання цих патернів дозволяє створювати масштабовані, гнучкі та підтримувані програмні системи.
##  Загальний висновок ( 19-20)
Структурні патерни допомагають керувати складністю архітектури.
- Composite - робота з ієрархіями  
- Decorator - розширення поведінки  
- Adapter - інтеграція  
- Facade - спрощення  
- Proxy - контроль доступу  
Їх комбіноване використання є основою сучасного проєктування програмних систем.
