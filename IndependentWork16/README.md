# Самостійна робота №16  
## Тема: Схема розподілу відповідальностей модуля (SRP)
### Мета роботи
Навчитися застосовувати принцип єдиної відповідальності (SRP) для декомпозиції складного
модуля, а також використовувати UML-діаграми для візуалізації структури системи.
## Опис завдання
У роботі було реалізовано сервіс прогнозу погоди (WeatherForecastService).
Спочатку створено "поганий" клас, який порушує SRP, виконуючи одразу кілька функцій:
- отримання даних з API;
- парсинг;
- збереження;
- відображення.
Після цього виконано рефакторинг з використанням:
- принципу єдиної відповідальності (SRP);
- принципу інверсії залежностей (DIP);
- інтерфейсів та їх реалізацій.
## Реалізовані компоненти
- IWeatherApiClient — отримання даних
- IWeatherParser — парсинг даних
- IWeatherRepository — збереження
- IWeatherDisplay — відображення
- WeatherService — координація роботи компонентів
## UML-діаграма
Діаграма класів створена за допомогою PlantUML та відображає всі інтерфейси, реалізації
та залежності між ними.
## Контрольні питання
### 1. Що таке SRP?
Принцип єдиної відповідальності (SRP) означає, що кожен клас повинен мати лише одну
причину для зміни. Це підвищує читабельність коду, спрощує тестування та супровід системи.
### 2. Що таке God Object?
God Object — це клас, який бере на себе надто багато відповідальностей.
Він порушує SRP, стає складним для підтримки та змін.
### 3. Як декомпозиція допомагає дотримуватися SRP?
Декомпозиція дозволяє розбити складний клас на кілька простих,
кожен з яких виконує одну чітку функцію.
Наприклад, у цій роботі один клас було розбито на сервіси отримання,
парсингу, збереження та відображення даних.
### 4. Навіщо потрібні UML-діаграми?
UML-діаграми дозволяють візуалізувати структуру системи,
швидко зрозуміти відповідальності класів та звʼязки між ними.
