# Factory Method vs Abstract Factory
## Опис
Цей репозиторій містить теоретичний матеріал та пояснення щодо породжувальних патернів проєктування:
1. Factory Method  
2. Abstract Factory  

Мета — показати різницю між цими патернами, їх переваги, недоліки та сценарії використання.

## Factory Method

### Сутність
Factory Method визначає інтерфейс для створення одного продукту, дозволяючи підкласам вирішувати, який саме клас інстанціювати.

### Коли використовувати
1. Коли система працює з одним типом продукту, але з різними реалізаціями.
2. Коли базовий клас не повинен знати про конкретні класи, які створюються.
3. Коли потрібно легко розширювати систему новими продуктами.

### Переваги
1. Відповідає принципу відкритості/закритості (OCP).
2. Зменшує зв’язаність між клієнтом і конкретними класами.
3. Простота реалізації.

### Недоліки
1. Збільшує кількість класів.
2. Підходить лише для одного типу продукту.


## Abstract Factory

### Сутність
Abstract Factory надає інтерфейс для створення сімейств взаємопов’язаних об’єктів без прив’язки до конкретних класів.

### Коли використовувати
1. Коли потрібно створювати групи узгоджених між собою об’єктів.
2. Коли система повинна підтримувати різні платформи або теми оформлення.
3. Коли важлива сумісність між створюваними об’єктами.

### Переваги
1. Гарантує узгодженість між продуктами однієї родини.
2. Дозволяє легко замінювати цілі сімейства продуктів.
3. Зменшує залежність від конкретних реалізацій.

### Недоліки
1. Більш складна архітектура.
2. Складно додавати нові типи продуктів.

## Порівняння
Factory Method застосовується для створення одного логічного типу продукту, але з різними конкретними реалізаціями. Клієнтський код працює з абстракцією (інтерфейсом або базовим класом), а вибір конкретної реалізації делегується підкласам-фабрикам. Це дозволяє легко розширювати систему новими типами продуктів без зміни коду, який їх використовує. Патерн добре підходить для локальних задач створення об’єктів, де не потрібна координація між кількома різними продуктами.

Abstract Factory використовується тоді, коли система повинна створювати не окремі об’єкти, а набори взаємопов’язаних продуктів, які мають бути узгодженими між собою. Заміна конкретної фабрики призводить до заміни всієї “родини” об’єктів (наприклад, компонентів інтерфейсу для різних платформ). Це спрощує підтримку багатоваріантних конфігурацій системи, але ускладнює архітектуру та робить менш зручним додавання нових типів продуктів.

## Висновок
Factory Method доцільно використовувати, коли потрібно створювати один тип об’єктів з різними реалізаціями.  
Abstract Factory варто застосовувати, коли система працює з цілими наборами взаємопов’язаних об’єктів.
