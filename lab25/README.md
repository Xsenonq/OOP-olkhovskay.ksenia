# Lab 25  
## Інтеграція патернів: Factory Method + Singleton + Strategy + Observer
##  Мета роботи
Розробити інтегровану систему, що демонструє взаємодію між компонентами, які реалізують патерни:
- Factory Method
- Singleton
- Strategy
- Observer
та перевірити коректність їхньої спільної роботи в різних сценаріях.
#  Архітектура системи
Система складається з трьох основних підсистем:
1. Logging (Factory Method + Singleton)
2. Data Processing (Strategy)
3. Event Notification (Observer)
Взаємодія між ними демонструється в методі Main.
#  1. Factory Method
##  Призначення
Factory Method дозволяє створювати об’єкти без жорсткої прив’язки до їх конкретних класів.
##  Реалізація
Інтерфейс: ILogger
Конкретні реалізації:
- ConsoleLogger
- FileLogger
Абстрактна фабрика:LoggerFactory
Конкретні фабрики:
- ConsoleLoggerFactory
- FileLoggerFactory
##  Результат
LoggerManager може отримувати різні типи логерів без зміни власного коду.
#  2. Singleton
##  Призначення
Забезпечити існування лише одного екземпляра менеджера логування.
## Реалізація
Клас:LoggerManager
Особливості:
- Приватний конструктор
- Статичний екземпляр
- Метод Initialize()
- Властивість Instance
## Результат
Уся система використовує єдиний центр логування.
#  3. Strategy
##  Призначення
Дозволити динамічно змінювати алгоритм обробки даних під час виконання програми.
##  Реалізація
Інтерфейс: IDataProcessorStrategy
Конкретні стратегії:
- EncryptDataStrategy
- CompressDataStrategy
Контекст:DataContext
Метод: SetStrategy()
##  Результат
Алгоритм обробки можна змінювати без модифікації клієнтського коду.
#  4. Observer
##  Призначення
Автоматично повідомляти підписників про завершення обробки даних.
##  Реалізація
Subject: DataPublisher
Подія: event Action<string> DataProcessed
Observer:ProcessingLoggerObserver
##  Результат
Після обробки даних автоматично викликається логування через Singleton.
#  Демонстраційні сценарії
##  Сценарій 1: Повна інтеграція
- Ініціалізація LoggerManager з ConsoleLoggerFactory
- Використання EncryptDataStrategy
- Підписка ProcessingLoggerObserver
- Обробка даних
- Публікація події
 Результат: логування відображається в консолі.
##  Сценарій 2: Динамічна зміна логера
- Після першої обробки викликається:
LoggerManager.Instance.ChangeFactory(new FileLoggerFactory());
- Повторна обробка даних
##  Сценарій 3: Динамічна зміна стратегії
-Викликається:
context.SetStrategy(new CompressDataStrategy());
-Повторна обробка даних
 Результат: дані обробляються новим алгоритмом.
#  Висновок

Реалізовано Singleton для централізованого керування логуванням.
Реалізовано Strategy для динамічної зміни алгоритмів.
Реалізовано Observer для автоматичного сповіщення про обробку даних.
