# Лабораторна робота №22 (коротко про лабу 22)
## Тема
LSP: виявлення порушень і альтернативи
## Варіант
2. Птахи (Bird & Penguin)
## Мета роботи
Поглибити розуміння принципу підстановки LSP,
навчитися ідентифікувати його порушення в ієрархіях класів
та застосовувати альтернативні архітектурні підходи
для створення LSP-сумісних рішень.
## Короткий опис роботи
У роботі реалізовано приклад ієрархії класів, що порушує
принцип підстановки Лісков, та продемонстровано наслідки
цього порушення у клієнтському коді.
Запропоновано альтернативне рішення шляхом зміни ієрархії
та використання інтерфейсів, яке повністю дотримується LSP.
## Результат
Проєкт компілюється та запускається без помилок.
Клієнтський код коректно працює з новою архітектурою,
що підтверджує дотримання принципу LSP.
# ЗВІТ (Чітко про виконану роботу в цій лабораторній)
## 1. Теоретичні відомості
Принцип підстановки LSP
є одним з п’яти принципів SOLID та формулюється наступним чином:
1. Об’єкти похідного класу повинні мати можливість замінювати
2. об’єкти базового класу без порушення коректної роботи програми.
Це означає, що похідний клас не має права:посилювати передумови (preconditions),послаблювати постумови (postconditions),змінювати очікувану поведінку базового класу.
## 2. Опис початкової ієрархії (порушення LSP)
Було створено базовий клас `Bird`, який містить метод `Fly()`.
Цей метод задає контракт: будь-який об’єкт типу `Bird`
повинен мати можливість виконати політ.
Похідний клас `Penguin` перевизначає метод `Fly()`,
викидаючи виняток `NotImplementedException`, оскільки
пінгвіни не здатні літати.
## 3. Аналіз порушення принципу LSP
Клієнтський код працює з базовим типом `Bird` та викликає метод `Fly()`,
очікуючи коректного виконання.
При підстановці об’єкта типу `Penguin` замість `Bird`
відбувається аварійне завершення програми, що свідчить про
порушення принципу підстановки Лісков.
Похідний клас порушує контракт базового класу,
оскільки змінює гарантовану поведінку методу.
## 4. Альтернативне рішення (дотримання LSP)
Для усунення проблеми було застосовано підхід
зміни ієрархії класів.
Абстрактний клас `Bird` тепер описує лише загальну
поведінку птаха — рух (`Move()`).
Поведінку польоту винесено в окремий інтерфейс `IFlyable`,
який реалізують лише ті класи, що дійсно можуть літати.
Таким чином, клас `Penguin` більше не зобов’язаний
реалізовувати некоректний для нього метод.
## 5. Демонстрація коректної роботи
Оновлений клієнтський код працює з відповідними абстракціями
та викликає лише ті методи, які гарантовано підтримуються
конкретним об’єктом.
Програма виконується без помилок, що підтверджує
дотримання принципу LSP.
## 6. Висновки
У ході виконання лабораторної роботи було досліджено
типове порушення принципу підстановки LSP.
Застосування зміни ієрархії та інтерфейсів дозволило
усунути логічну помилку та зробити систему більш гнучкою,
надійною і розширюваною.
Дотримання LSP є необхідною умовою для створення
якісного об’єктно-орієнтованого програмного забезпечення.
